# TASKS — разбиение работ на параллельные задачи

Документ для управления разработкой MVP по спринтам.

## Принципы
- Каждая задача имеет `ID`, `Owner`, `Depends on`, `Definition of Done`.
- Параллелим максимум: Android UI/камера, OCR-парсер, Data/экспорт.
- Инкременты должны заканчиваться рабочим APK.

---

## Этап 0. Подготовка (foundation)

### T0-1. Спецификация форматов полей
- **Owner**: Data/OCR
- **Depends on**: —
- **Суть**: зафиксировать правила нормализации `price`, `price_per_kg`, `weight_volume`.
- **DoD**:
  - JSON/markdown-спека с примерами «сырой OCR → нормализованная запись»;
  - покрыты форматы `г/кг/мл/л`, дробные и целые цены.

### T0-2. Набор тестовых изображений
- **Owner**: Product/Data
- **Depends on**: —
- **Суть**: собрать 80–150 фото из 1–2 магазинов.
- **DoD**:
  - папка датасета структурирована;
  - минимум 50 «хороших» и 30 «сложных» кадров.

### T0-3. Метрика качества
- **Owner**: Data/OCR
- **Depends on**: T0-1, T0-2
- **Суть**: определить KPI на поле и на запись.
- **DoD**:
  - формулы `field_accuracy` и `record_accuracy`;
  - baseline-отчёт для первой версии пайплайна.

---

## Этап 1. Базовый OCR pipeline (MVP core)

### T1-1. Android каркас приложения
- **Owner**: Android
- **Depends on**: —
- **Суть**: создать app-модуль, навигацию экранов, базовые permissions.
- **DoD**:
  - запуск на устройстве Android 13;
  - экран Camera Preview + экран Confirm/Edit + экран History.

### T1-2. Интеграция CameraX + снимок кадра
- **Owner**: Android
- **Depends on**: T1-1
- **Суть**: захват кадра и передача bitmap в OCR-пайплайн.
- **DoD**:
  - стабильный захват кадров без падений;
  - ограничение: один ценник в кадре (визуальная подсказка рамкой).

### T1-3. OCR слой (ML Kit)
- **Owner**: OCR
- **Depends on**: T1-2
- **Суть**: извлекать полный текст и bounding boxes.
- **DoD**:
  - возвращается структурированный результат OCR;
  - лог confidence и latency на устройстве.

### T1-4. Парсер полей + валидация
- **Owner**: OCR/Data
- **Depends on**: T0-1, T1-3
- **Суть**: извлечение 4 полей и проверка диапазонов.
- **DoD**:
  - парсинг названия/цены/цены за кг/веса-объёма;
  - правило округления спорной дробной части вверх;
  - валидация (`price < 100000`).

### T1-5. Экран подтверждения/ручной правки
- **Owner**: Android
- **Depends on**: T1-4
- **Суть**: пользователь редактирует поля и подтверждает запись.
- **DoD**:
  - редактируемые поля с первичной автоподстановкой;
  - кнопка «Сохранить запись».

## Этап 2. Live-режим и сессии

### T2-1. Потоковый анализ кадров
- **Owner**: Android/OCR
- **Depends on**: T1-3
- **Суть**: обработка каждого N-го кадра (1–3 сек/распознавание).
- **DoD**:
  - ограничение частоты анализа;
  - защита от параллельных OCR-запусков.

### T2-2. Стабилизация результата
- **Owner**: OCR
- **Depends on**: T2-1, T1-4
- **Суть**: подтверждать результат только при 2–3 схожих подряд распознаваниях.
- **DoD**:
  - уменьшено число ложных срабатываний;
  - вывод confidence score в UI.

### T2-3. Логика сессий сканирования
- **Owner**: Data/Android
- **Depends on**: T1-5
- **Суть**: каждый запуск live-режима создаёт новую `scan_session`.
- **DoD**:
  - записи привязаны к session_id;
  - старт/стоп сессии корректно сохраняются.

### T2-4. Политика сохранения
- **Owner**: Android
- **Depends on**: T2-2, T1-5
- **Суть**: в live-режиме сохранять **только после подтверждения**.
- **DoD**:
  - нет автосохранения без действия пользователя.

---

## Этап 3. Данные и экспорт

### T3-1. Room schema + DAO
- **Owner**: Data/Android
- **Depends on**: T2-3
- **Суть**: таблицы `scan_session`, `scan_item`.
- **DoD**:
  - миграции и индексы;
  - CRUD без ошибок.

### T3-2. Экран мастер-таблицы
- **Owner**: Android
- **Depends on**: T3-1
- **Суть**: список всех записей и фильтр по сессии.
- **DoD**:
  - отображение всех полей;
  - фильтрация по сессии/дате.

### T3-3. CSV экспорт
- **Owner**: Data/Android
- **Depends on**: T3-1
- **Суть**: экспорт в UTF-8 CSV с разделителем `;`.
- **DoD**:
  - экспорт всей базы;
  - экспорт выбранной сессии;
  - файл открывается в Excel/Google Sheets без поломки кириллицы.

---

## Этап 4. Улучшение качества (опционально)

### T4-1. Анализ ошибок OCR/парсинга
- **Owner**: Data/OCR
- **Depends on**: T0-3, T1-4
- **Суть**: выделить топ-ошибки (блики, ракурс, шрифты).
- **DoD**:
  - отчёт с ранжированием причин ошибок;
  - список приоритетных улучшений.

### T4-2. Дообучение детектора ценника
- **Owner**: ML (Python)
- **Depends on**: T4-1
- **Суть**: обучить лёгкую модель детекции (YOLOn) и сравнить с baseline.
- **DoD**:
  - метрики до/после;
  - экспорт модели для mobile inference.

---

## Параллельный план первого спринта (пример)

- **Поток A (Android)**: T1-1 → T1-2 → T1-5
- **Поток B (OCR/Data)**: T0-1 → T1-3 → T1-4
- **Поток C (Dataset/QA)**: T0-2 → T0-3

Цель спринта: в конце получить APK, где можно сделать фото ценника, распознать 4 поля, поправить вручную и сохранить запись.
